## F. Interview Prep

**Role archetype:** Data Infrastructure / Backend Engineer. Daris is moderate strength here (strong fundamentals, no domain experience). 5 recommended stories below.

### Story 1. Data Pipeline at Scale (Mini-Stockfish)
**Situation:** Train neural network (NNUE) on 316M chess positions. Naive loading would OOM and be slow.
**Task:** Design data pipeline for GPU training at scale.
**Action:** Used PyTorch DataLoaders with memory mapping (mmap) to avoid loading full dataset into RAM. Implemented multi-worker sampling with shuffle. Benchmarked batch sizes (64, 128, 256, 512). Profiled I/O bottleneck (disk throughput limited to 500MB/s).
**Result:** Trained efficiently. Competitive play strength (high Elo). Completed training in reasonable time.
**Reflection:** Data loading is often the bottleneck, not compute. Understanding I/O patterns and storage systems enables scaling.

### Story 2. ETL Pipeline (FishSense)
**Situation:** Extract biological metrics (fish length, mass) from underwater videos. Need to ingest 1000s of videos, extract frames, annotate, generate training data.
**Task:** Build Python ETL to standardize data ingestion for new researchers.
**Action:** Wrote pipeline (OpenCV video ingestion, frame extraction at regular intervals, Detectron2 annotation, curation). Stored intermediate results on disk (checkpointing). Made resumable (crash tolerance).
**Result:** 40% faster training setup for new researchers. Increased reproducibility across team.
**Reflection:** Good ETL enables downstream work. Checkpointing and resumability are underrated.

### Story 3. Query Optimization (PatentIQ)
**Situation:** Patent search needed to query 12k+ documents by relevance + filter by inventor, patent class, date. Query latency was 5 seconds.
**Task:** Design efficient database queries. Push filtering down from application layer to database.
**Action:** Analyzed query patterns (most common filters). Added indexes on (patent_class, date, inventor). Implemented pgvector approximate nearest neighbor search (HNSW index). Built recursive SQL query builder for dynamic filters.
**Result:** Latency improved from 5s to 200ms (p99). Reduced application memory.
**Reflection:** Databases are query engines. Query patterns drive schema design. Good indexing is infrastructure.

### Story 4. Caching Strategy (Groundwork Books)
**Situation:** Real-time inventory sync with Square API. 50+ concurrent requests to check stock. API latency was 2.1s per request.
**Task:** Design two-tier caching to reduce latency while maintaining correctness during syncs.
**Action:** Built Redis look-aside cache for hot inventory data (TTL 5 minutes). Implemented client-side request coalescing (batch 50+ concurrent lookups into single API call, 2-minute TTL). Invalidated cache on inventory update events.
**Result:** p99 latency reduced to 180ms. API calls reduced by 60%.
**Reflection:** Caching strategy compounds benefits. Application-level caching + database caching + CDN caching = scalability.

### Story 5. Infrastructure Observability & Alerting (Implicit in PatentIQ Deployment)
**Situation:** PatentIQ running on Vercel (frontend) and EC2 (backend). Need to know when things break.
**Task:** Set up monitoring, logs, alerts.
**Action:** Configured CloudWatch for EC2 (CPU, memory, disk). Set up application logs (stderr to CloudWatch). Created alarms for high error rates. Added health checks to Vercel.
**Result:** Caught and fixed 2 production issues before users noticed. Uptime >99.9%.
**Reflection:** Observability is invisible until you need it. Good logging saves hours of debugging.
