## F) Interview Prep

### 6 STAR Stories (Mapped to JD)

#### Story 1: PatentIQ — End-to-End Agentic Patent Research Agent

**JD Match:** Multi-agent coordination, tool orchestration, multi-step workflows

- **Situation:** Accelerator cohort needed rapid MVP of patent search platform. Non-lawyer founders struggled with expensive, slow, opaque patent research. 4-month timeline; 14-person team.
- **Task:** Architect full-stack AI platform combining semantic search, LLM recommendations, document upload analysis, CPC classification. Coordinate 3 external APIs. Ship something real.
- **Action:** Designed **multi-step agentic orchestration**: (1) Parse user query (semantic intent extraction), (2) pgvector similarity ranking over 5M+ USPTO patents (OpenAI embeddings), (3) Claude recommendation engine (structured JSON output for reasoning), (4) Document upload analysis + CPC suggestion (another Claude call with context injection), (5) LRU cache optimization + batch processing for latency. Implemented structured error logging for debugging. Built fallbacks (if semantic fails, keyword search; if Claude timeout, queue for async).
- **Result:** Shipped working MVP. Early-stage founders actively using for real patent research. GitHub public. Demonstrates end-to-end ownership, multi-step orchestration, API coordination, observability mindset.

#### Story 2: Groundwork Books — Production Caching & Multi-API Orchestration

**JD Match:** Production resilience, observability, multi-API coordination

- **Situation:** Live e-commerce bookstore needed real-time inventory sync across 4,000+ SKUs. Square API calls per lookup were slow (1–3s latency); killing UX. No engineers on staff; I was sole technical operator.
- **Task:** Reduce latency to sub-200ms p99 without rewriting core system. Sync must stay real-time (inventory changes within minutes).
- **Action:** Two-tier caching architecture: (1) **Look-aside Redis** caching Square API responses, invalidated on inventory updates (sub-200ms lookup). (2) **Client-side request coalescing hook** batching 50+ concurrent inventory checks into single API call with 2-min TTL. Added structured logging + Sentry error tracking for production debugging. Built graceful degradation: if cache stale, show best estimate rather than hanging.
- **Result:** 90% latency reduction (1–3s → <200ms p99). Site lives on Vercel. Real users, real money. Demonstrates: resilience thinking, multi-API coordination, observability + monitoring, pragmatic tradeoffs.

#### Story 3: FlipperZillow — 48-Hour Integration of 5 Unfamiliar GPU Models

**JD Match:** Adaptability, rapid iteration under pressure, multi-step orchestration

- **Situation:** 24-hour hackathon. Goal: build immersive 3D property tours with AI narration. Challenge: unfamiliar AMD ROCm GPU cloud, 5 pre-trained models (DepthAnythingV2, DFormerV2, SAM3D Object, Claude, ElevenLabs), zero prior experience on this stack.
- **Task:** Ship working product in 24 hours. No time for "learning phase."
- **Action:** Reverse-engineered AMD ROCm setup + CUDA↔ROCm compat issues. Built parallel GPU pipelines: (1) DepthAnythingV2 for depth maps from property images, (2) DFormerV2 for semantic segmentation, (3) SAM3D for 3D object extraction (point clouds). Assembled cloud workflow: fetch Realtor.com photos → process through dual GPU pipeline → output .glb meshes. (4) Claude API for semantic analysis → professional realtor scripts. (5) ElevenLabs for narration. Integrated Google Maps + WebSpatial for Vision Pro experience; Three.js fallback for web.
- **Result:** Shipped working product. Immersive 3D tours. Demonstrates: rapid learning (unfamiliar stack in 2 days), multi-step system design, comfort with complex orchestration, pragmatic tradeoffs (WebSpatial vs fallback).

#### Story 4: Simplerfy — Structured Prompt Engineering for Job Analysis

**JD Match:** LLM APIs, prompt engineering, structured output reliability

- **Situation:** Job candidates spend hours rewriting resumes for each application. Needed AI system to automatically analyze job descriptions, extract requirements, and generate tailored resumes **reliably** (not hallucinating skills candidates don't have).
- **Task:** Build SaaS combining Claude API + structured output + LaTeX templating. Key constraint: 1-page resume no matter what; no adding fake skills.
- **Action:** Designed Claude prompts with **strict structured output** (JSON schema): extract skills → category → criticality. Prompt engineering: few-shot examples of good/bad extractions; explicit guardrails ("never add skills candidate doesn't have"). Implemented resume scoring system comparing candidate background to JD requirements. Built LaTeX template management + real-time PDF compilation. Chrome Extension for 1-click job board integration.
- **Result:** Working SaaS. Chrome Extension published. Handles 15+ job board formats. Demonstrates: prompt design for reliability, structured output thinking, integration (Chrome API + backend), end-to-end feature ownership.

#### Story 5: Mini-Stockfish Chess Engine — Full-Stack System from ISA to GPU Optimization

**JD Match:** Ship independently, influence architecture, range across abstraction levels

- **Situation:** Wanted to understand chess engine architecture + neural networks + performance optimization. Built entire system solo: ISA design → neural network training → GPU optimization → GUI.
- **Task:** Competitive chess engine. Full ownership of all layers.
- **Action:** Architected HalfKP NNUE neural network trained on 316M Stockfish positions. Built alpha-beta search with killer moves, late-move reduction (LMR), quiescence search, transposition tables. **Optimization obsession:** Profiled bottleneck (leaf node evaluation: 1.1ms). Rewrote in Cython + C++. Optimized move generation with bitboards + Zobrist hashing. Reduced evaluation to 0.25ms (77% speedup). Endgame databases (Syzygy/Gaviota). PyGame GUI.
- **Result:** Competitive chess engine. GitHub public. Demonstrates: full-stack ownership (from algorithms to hardware optimization), architectural decision-making, performance-driven thinking, range across abstractions.

#### Story 6: PatentIQ Product Design — Intuitive UI Without Designer

**JD Match:** Product sensibility, ship without dependencies, UI for non-designers

- **Situation:** Patent search interface needed to be intuitive for non-technical founders and lawyers. No designer on team. Needed to ship fast and usably.
- **Task:** Design + iterate on UX in parallel with backend development.
- **Action:** Iterated on search experience: search input with instant semantic results (no submit button = immediate feedback), results page with relevance scores + keyword highlighting, one-click patent details, AI-generated drafting suggestions, document upload for CPC auto-suggestion. Used Tailwind CSS for speed. Gathered founder feedback (daily stand-ups) and iterated rapidly (results layout, terminology, interaction patterns). Tracked adoption metrics.
- **Result:** 15% adoption lift after UI simplification. Demonstrates: product judgment (user-focused thinking), iteration speed (feedback → code → deploy in hours), metrics-driven (not just building, measuring impact).
