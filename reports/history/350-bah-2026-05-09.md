**Situation:** Joined PatentIQ as lead engineer during a 4-month accelerator. 14-person team, product manager accelerator. Goal: ship a patent search MVP.

**Task:** Architect the entire system from JD requirements to launch. Balance between speed (ship in 6 weeks) and robustness (early-stage founders trust the search).

**Action:**
- Proposed PostgreSQL + pgvector over expensive vector DB (Pinecone, Weaviate)
- Built hybrid ranking system combining semantic similarity with keyword matching

- Deployed to Supabase + Vercel with Auth0 auth

**Result:** Shipped working MVP in 6 weeks. Early users (founders) doing professional-grade patent research on the platform.

**Reflection:** End-to-end ownership means caring about the whole system, not just your layer. I made database decisions that affected API performance, which affected frontend perceived speed. That systems-level thinking is what the "full-stack" part really means.

### Story 4: Technical Judgment (PostgreSQL Decision)

**Situation:** PatentIQ architecture design. Team proposed three options: MongoDB + Pinecone, Weaviate (managed), PostgreSQL + pgvector (open-source).

**Task:** Recommend database approach for MVP.

**Action:**
- Analyzed cost trade-off: Pinecone ($200+/month), Weaviate (managed SaaS), PostgreSQL (free, already running Supabase)
- Proposed PostgreSQL + pgvector for MVP (lower cost, easier iteration)
- Data scientist and backend engineer skeptical (Pinecone is production-tested)
- Presented benchmarks: pgvector performance was sufficient for initial dataset size (~100K docs)
- Proposed: "Use pgvector for MVP, migrate to Pinecone if we hit scale limitations"

**Result:** Team agreed. Shipped MVP faster, saved $10K+ in infrastructure costs.

**Reflection:** Good technical judgment isn't about always picking the newest tool. It's about matching tool complexity to problem size and team capacity.
