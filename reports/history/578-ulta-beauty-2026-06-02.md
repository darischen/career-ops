## Block F — Interview Prep

### Story 1: Built LLM Orchestration Under Deadline (FlipperZillow)

**Situation:** 24-hour hackathon. Build AI house tour platform with 3D visualization using unfamiliar tech.

**Task:** Orchestrate 5+ pre-trained models (DepthAnythingV2, DFormerV2, SAM3D, Google Maps, Claude Sonnet) into a coherent pipeline.

**Action:**
- Researched model APIs in parallel (while spinning up GPU environment)
- Built inference pipeline: image input → depth estimation → 3D reconstruction → Claude narration
- Handled errors: SAM3D broke on ROCm, pivoted to CUDA
- Optimized latency: batch images, cache model weights, stream output

**Result:** Working 3D house tours with professional AI narration (Claude Sonnet 4.6), delivered in 24 hours.

**Reflection:** "Orchestrating models is like orchestrating a supply chain: each step has latency, errors can cascade, and bottlenecks compound. I think in pipelines. When SAM3D failed, I didn't try to patch it; I rerouted the pipeline around it. That's how you ship under pressure."

---

### Story 2: Shipped Production RAG System (PatentIQ)

**Situation:** Build semantic patent search for 8M USPTO patents. Need both keyword + semantic understanding.

**Task:** Design and implement RAG (retrieval-augmented generation) pipeline for patent analysis.

**Action:**
- Implemented two-stage search: keyword (BM25) → semantic re-rank (pgvector cosine similarity)
- Chunked patents by section (claims, abstract, description) for fine-grained search
- Built recursive SQL for patent family queries (hierarchical IPC classification)
- Integrated Claude API to analyze retrieval results, generate summaries

**Result:** Semantic search 40% better than keyword-only. 80ms query latency on 8M dataset.

**Reflection:** "RAG is about understanding the entire flow: what data to retrieve, how to chunk it, how to rank candidates, how to use LLM context wisely. I thought about each step independently, then optimized the seams (reranking, context window management). That systems thinking is what makes RAG work at scale."

---

### Story 3: Integrated Multiple APIs into One System (PatentIQ)

**Situation:** PatentIQ has 4+ external dependencies (Claude API, OpenAI embeddings, Auth0, Supabase, Square for payments).

**Task:** Integrate all of them into one coherent backend without creating a spaghetti mess.

**Action:**
- Built abstraction layer: separate services for LLM calls (Claude + OpenAI), auth (Auth0), database (Supabase)
- Implemented retry logic for API failures (exponential backoff, circuit breaker)
- Added monitoring: which APIs are slow, which are failing
- Coordinated with team: data scientist owns embedding strategy, backend engineer owns rate limiting

**Result:** Seamless integration. API failures are graceful. System can handle 100+ concurrent users.

**Reflection:** "Tool integration isn't just gluing APIs together. It's understanding each tool's failure modes (rate limits, timeouts, auth expiry) and building systems that handle them. I do this by thinking of each API as a state machine: what can go wrong? How do I recover?"

---

### Story 4: Learned Retail Operational Problem Space (Groundwork Books)

**Situation:** Groundwork sells 4,000+ SKUs. Real-time inventory is critical (if SKU sells, other stores need to know immediately).

**Task:** Design inventory system that's real-time but doesn't overwhelm Square API.

**Action:**
- Interviewed store managers: why do you care about inventory updates? (Don't want to oversell, need to restock intelligently)
- Analyzed bottleneck: Square API rate limits + slow lookup latency
- Implemented caching: Redis TTL 2 minutes = updated 30x per day, sufficient for retail ops
- Added alerts: when SKU inventory drops below threshold, notify inventory manager

**Result:** Real-time enough for operations (30-minute eventual consistency), no overselling, store team happy.

**Reflection:** "Understanding the business problem comes before the tech solution. Store managers don't care if inventory is 'real-time' in the computer science sense (milliseconds). They care if it's 'real-time' in the business sense (before they make a decision). That's a different threshold. I always ask: what decision is this data supporting? What's the acceptable latency for that decision?"

---

### Story 5: Self-Driven Learning & Pragmatism (Overall career narrative)

**Situation:** Throughout PatentIQ, Groundwork, FlipperZillow, encountered 10+ new technologies (pgvector, ROCm, DepthAnythingV2, Square API, Auth0, etc.).

**Task:** Learn on the job without blocking shipping.

**Action:**
- For each new tech, asked: "What problem does this solve?" before diving into docs
- Learned by doing: sketched out solution, then read docs for optimization
- When learning hit diminishing returns (docs vs. experimentation), stopped reading and started coding
- Flagged unknowns proactively: "I haven't used Auth0 before. It might take 2-3 days to integrate. Want to proceed?"

**Result:** Shipped products with continuous learning. No "analysis paralysis."

**Reflection:** "Learning is a continuous process, not a prerequisite. I don't need to know everything before I start. I learn iteratively: understand the problem, sketch a solution, implement, discover edge cases, iterate. That's how real work happens."
