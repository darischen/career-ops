## Block F — Interview Prep: STAR+R Stories

Score is 3.8 → **Prepare these 5 stories:**

### Story 1: Shipped Product Under Pressure (Architecture & Speed)

**Situation:**  
PatentIQ, 4-month Product Manager Accelerator, 14-member team (PMs, designers, engineers), building an AI patent search MVP from scratch.

**Task:**  
Lead architect + primary coder responsible for deciding database architecture, semantic search strategy, and shipping a working prototype in 6 weeks while mentoring other engineers on decisions.

**Action:**
- Proposed PostgreSQL + pgvector for semantic search instead of Pinecone (new vendor cost, complexity). 
- Made the case: Postgres already running, pgvector is open-source, faster iteration for MVP.
- Collaborated with data scientist and backend engineer to validate benchmarks.
- Built the recursive SQL query builder with Zod validation to handle complex filtering.
- Deployed to Vercel (frontend) + EC2 (FastAPI backend) + Supabase (database + auth via Auth0).

**Result:**  
Working patent search MVP shipped in 6 weeks. Semantic latency reduced from 15s (naive) to <2s. Hybrid ranking (70% semantic, 30% keyword). Team shipped without rework on database layer.

**Reflection:**  
Learned that technical credibility comes from collaborating with peers (asking them to research independently, then deciding together) rather than overriding them. Also learned that choosing "good enough" architecture (Postgres instead of new vendor) unblocks the team faster than optimizing for perfection.

---

### Story 2: Debugged Performance at Scale (Caching & Optimization)

**Situation:**  
Groundwork Books, production e-commerce platform with 4,000+ book SKUs. Square POS API was being hammered with inventory checks, causing p99 latency of 2.1 seconds. Customer experience suffering.

**Task:**  
Architect a caching layer to reduce API latency without losing real-time accuracy.

**Action:**
- Designed two-tier caching: Redis look-aside cache (1-minute TTL for most items) + client-side request coalescing hook (2-minute cache, batches 50+ concurrent requests into 1 API call).
- Analyzed Square API limits and request patterns to set TTL appropriately (hot items cached longer, cold items refreshed sooner).
- Implemented client-side hook to detect duplicate in-flight requests and dedup them locally.

**Result:**  
p99 latency dropped from 2.1s to 380ms (82% reduction). API call volume reduced by ~90%. Real-time inventory still accurate (cache invalidation on manual updates). Site performance jumped from slow to fast.

**Reflection:**  
Performance work isn't just about throwing caches at the problem. Understanding the actual request pattern (many concurrent identical requests) was key. The coalescing hook was the bigger win than Redis.

FlipperZillow 24-hour hackathon. Goal: Build AI-powered 3D house tours with neural networks. Tech stack: ROCm (AMD GPU), DepthAnythingV2, DFormerV2, SAM3D (all unfamiliar).

**Result:**

**Reflection:**

### Story 4: Technical Leadership at Scale (Delegation & Mentorship)

**Situation:**  
PatentIQ team, 14 members across multiple disciplines (data scientist, backend engineer, frontend developer, multiple PMs, designers). Serving as lead architect and primary contributor.

**Task:**  
Coordinate work so that each person owns a clear piece without rework or blocking dependencies.

**Action:**
- Assigned end-to-end ownership: Data scientist owns ML pipeline (embeddings → ranking), backend engineer owns server integration + auth, frontend owns UI, PMs own roadmap.
- Tracked progress weekly, identified blocking dependencies early (e.g., "backend needs embeddings API by Thursday for frontend to integrate Friday").
- Served as lead architect but stayed hands-on (coding, not just reviewing). Participated in code reviews.
- When frontend developer struggled (new to Next.js), flagged it early to PMs and offered pair programming sessions (in progress, didn't get to finish but shows leadership intent).

**Result:**  
Shipped MVP with clear ownership, minimal rework, on schedule. 14-person team coordinated smoothly.

**Reflection:**  
Technical leadership isn't about being the smartest person in the room. It's about matching scope to skill and creating clarity about dependencies. The key was having weekly syncs and being ruthless about blocking issues.

---

### Story 5: Made a Trade-Off Decision (Pragmatism Over Perfection)

**Situation:**  
Mini-Stockfish chess engine neural network training. Original dataset: 13M positions. Requirements grew: 316M positions (24x growth).

**Task:**  
Keep the training pipeline running as dataset size increased beyond RAM.

**Action:**
- Original pipeline loaded entire dataset into memory (broke at 316M).
- Instead of just waiting for more RAM: rebuilt pipeline to stream from disk using PyTorch DataLoaders + memory-mapped files.
- Trade-off: Slower training iteration (can't fit full batch in memory), but now scalable.
- Result: Could train on 316M examples, achieve competitive play strength.

**Reflection:**  
Systems hit scaling limits. The answer isn't always "buy more hardware" — sometimes it's "change the architecture". Learning to recognize when a design is hitting its limit and redesign proactively (not after a crisis) saves months of work.
