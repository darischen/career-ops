## F. Interview Prep

**Role archetype:** Infrastructure / Backend Engineer. Daris is strong (but will need to emphasize scaling thinking). 5 recommended stories below.

### Story 1. Architecture for Scaling (PatentIQ Backend)
**Situation:** Patent search MVP needed to scale from 0 to 1000s of concurrent users. Single-server deployment would not work.
**Task:** Design backend architecture for scale.
**Action:** Used FastAPI (async, horizontal scalability). Deployed to EC2 with load balancing. Implemented caching layer (LRU cache in-process, Redis for cross-server state). Database: PostgreSQL with pgvector for distributed semantic search.
**Result:** Scaled to 1000s of concurrent searches. <200ms latency (p99).
**Reflection:** Scaling is about removing bottlenecks. Application layer (caching, async), database (indexes, replication), and infrastructure (load balancing) work together.

### Story 2. Developer Experience Focus (Personal Portfolio Site)
**Situation:** Wanted to share projects online. 100/100 Lighthouse target (performance + accessibility + SEO).
**Task:** Build site that developers respect (performance, clean code, responsive design).
**Action:** Custom SSG pipeline (Markdown to HTML at build time). Lazy loading, code splitting, edge caching. Responsive design. Dynamic OG image generation.
**Result:** 100/100 Lighthouse on all metrics. FCP <0.8s globally. Developers appreciate the attention to quality.
**Reflection:** Developer experience is a feature. Performance, reliability, and ease of use compound.

### Story 3. Database Architecture Decision (PatentIQ)
**Situation:** Semantic search requires dense vector storage. Team wanted MongoDB + Pinecone (managed services). Cost and iteration speed were concerns.
**Task:** Evaluate architectural tradeoffs (managed vs open-source).
**Action:** Built cost model for both approaches. Ran benchmarks on USPTO patent dataset (12k docs, 1536-dim embeddings). Estimated TCO: managed services $3K/month, PostgreSQL+pgvector $1K/month.
**Result:** Chose open-source. Shipped faster. Saved cost.
**Reflection:** Architecture decisions are cost + correctness + velocity tradeoffs. Understanding all three drives good decisions.

### Story 4. Performance Under Constraints (Mini-Stockfish Optimization)
**Situation:** Chess engine evaluating millions of positions per second. Hardware: commodity laptop. Budget: zero.
**Task:** Optimize critical path using only software techniques.
**Action:** Profiled (cProfile) to identify bottleneck (77% in leaf node evaluation). Rewrote hot path in Cython (10x) then C++ (5x more). Implemented Zobrist hashing (2x lookup hit rate). Incremental measurement.
**Result:** 77.2% latency reduction. Competitive play strength.
**Reflection:** For free optimization, profiling + incremental improvement beats rewriting everything.

### Story 5. Learning Enterprise Systems Fast (Implicit in PatentIQ Team Leadership)
**Situation:** Led 14-person team building production SaaS in 4 months. Had to learn Auth0, Vercel, EC2, Supabase, payment integrations simultaneously.
**Task:** Adopt new tools and services fast without slowing shipping.
**Action:** Delegated learning. Asked engineers to own tool evaluation (who owns Auth0? Who owns Vercel deployment?). Reviewed decisions but trusted team to learn.
**Result:** Shipped MVP on time. Each engineer mastered their domain.
**Reflection:** In enterprise environments, you'll learn many tools. Best approach: own one deep, understand others at surface level, delegate appropriately.
