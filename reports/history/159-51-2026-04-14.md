### Story 1. Data Pipeline Engineering (PatentIQ)
**Situation:** PatentIQ semantic search needed to rank 12k+ USPTO patents by relevance to user queries. Naive embedding search was slow.
**Task:** Design and implement efficient retrieval pipeline combining dense (embeddings) and sparse (keyword) signals.
**Action:** Used PostgreSQL pgvector for semantic search (OpenAI embeddings). Added SQL filtering (inventor, date range, technology class). Implemented LRU cache to avoid re-ranking same queries. Profiled query latency (initial 5s per query).
**Result:** Reduced latency to 200ms (p99). Scaled to handle 100+ concurrent searches without timeout.
**Reflection:** Data engineering is about tradeoffs: latency vs cost vs accuracy. Understanding your data (12k patents, 1536-dim embeddings) drives architecture.

### Story 2. Large-Scale Training (Mini-Stockfish)
**Situation:** Train neural network (NNUE) on 316M chess positions. Memory and I/O were bottlenecks.
**Task:** Optimize data pipeline for GPU training at scale.
**Action:** Used PyTorch DataLoaders with memory mapping (mmap) to avoid loading entire dataset into RAM. Implemented multi-worker sampling with shuffle. Benchmarked different batch sizes and number of workers.
**Result:** Trained 316M examples in reasonable time. Competitive play strength (high Elo rating).
**Reflection:** Data loading is the bottleneck in ML. Understanding I/O patterns and storage systems makes training practical.

### Story 3. SQL Query Optimization (Groundwork Books)
**Situation:** Inventory search had to query 4000+ SKUs in real-time while syncing with Square.
**Task:** Design schema and indexes to support fast keyword + semantic search without blocking inventory updates.
**Action:** Built Pinecone-like semantic layer using text embeddings. Indexed on (sku_id, category, embedding). Implemented write-ahead logging to handle concurrent inventory syncs.
**Result:** <100ms query latency, zero lost updates during sync.
**Reflection:** Schema design is the foundation of data systems. Choosing the right keys and indexes prevents performance regrets.

### Story 4. ETL Thinking (FishSense)
**Situation:** Detectron2 R-CNN model needed training data from 1000s of underwater videos.
**Task:** Build data pipeline: video ingestion, frame extraction, annotation, training dataset generation.
**Action:** Wrote Python ETL (OpenCV for frame extraction, Detectron2 for annotation, curation for quality). Ran on EC2 with checkpointing (resume if interrupted).
**Result:** 40% faster training setup for new researchers. Increased reproducibility.
**Reflection:** ETL pipelines enable everything downstream. Good data engineering multiplies model quality.

### Story 5. Profiling + Optimization (Mini-Stockfish)
**Situation:** Chess engine evaluating 1M positions took 1100ms. Needed <250ms for interactive play.
**Task:** Profile bottleneck and optimize critical path.
**Action:** Used cProfile to identify leaf node evaluation as 77% of runtime. Rewrote in Cython + C++ legal move generation. Implemented Zobrist hashing for faster position lookup.
**Result:** 77.2% latency reduction (1100ms to 250ms). Enabled real-time play.
**Reflection:** Profiling is non-negotiable. Never optimize without data.
