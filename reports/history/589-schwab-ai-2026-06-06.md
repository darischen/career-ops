### Story 2: Architected a System, Then Defended It (PatentIQ DB Choice)

**Situation:** PatentIQ team needed to decide on database architecture for semantic patent search over 12K+ documents.

**Task:** PMs proposed MongoDB, Pinecone, or Weaviate. I proposed PostgreSQL + pgvector. Need to convince stakeholders.

**Action:**
- Researched: Postgres + pgvector is open-source, zero new vendor cost, already running in production (Supabase), faster iteration than managed vector DB.
- Built a quick prototype: semantic search query on 1K patents in pgvector. Query time: <200ms.
- Presented to data scientist + backend engineer: "Here's the benchmark. Postgres is 3x cheaper, 2x faster iteration than Pinecone, and we can switch to Pinecone later if we need to scale to 10M+ documents."
- They came back a week later, agreed. We shipped with PostgreSQL + pgvector.

**Result:** Lower cost, faster iteration, same performance. Got technical buy-in without overriding anyone.

**Reflection:** Good architecture isn't about using the fanciest tool. It's about matching the tool to the problem size and timeline. I make technical decisions based on data, not hype.

---

### Story 3: Optimized for Real-World Performance (Groundwork Books 90% Latency Reduction)

**Situation:** Groundwork Books runs a live e-commerce platform. Square POS API was the bottleneck. Every product page load triggered 3-4 Square API calls. Result: 15-second page load. Users were bouncing.

**Task:** Cut latency to <200ms p99 without breaking real-time inventory.

**Action:**
- Analyzed the pattern: Square API is idempotent. Same query returns same result for ~2 minutes.
- Implemented Redis look-aside cache: Check cache first, fall back to API if miss, cache the result for 2 minutes.
- Added client-side request coalescing: If 50 concurrent users load the product page, batch their API requests into one.
- Tested: p99 latency dropped from 15s → 150ms.

**Result:** 90% latency reduction. Users no longer bounce. Conversion metrics improved.

**Reflection:** Performance isn't about fast code. It's about understanding the access pattern and matching the tool to the pattern. Caching strategies, batch processing, and async patterns are critical at scale.

**Situation:** PatentIQ accelerator: 14-person team (PMs, designers, engineers, data scientist). My role: architect the MVP, be primary technical contributor, and coordinate work.

**Task:** Ship AI-powered patent search MVP in 4 months. Complex integration: OpenAI embeddings, FastAPI backend, Next.js frontend, Auth0, Supabase, EC2.

**Action:**

- Scheduled weekly syncs to unblock dependencies (e.g., "backend API spec ready by Wednesday so frontend can start mocking").
- Stayed hands-on: I wrote core modules (semantic search, LRU cache, recursive SQL builder), reviewed all PRs, debugged integration issues.
- Matched complexity to skill: Gave junior designer high-visibility work (UI), paired them with senior designer for feedback loops.

**Result:** Shipped MVP on schedule. Clear ownership, minimal rework, team velocity improved every sprint.

**Reflection:** Leadership isn't management. It's removing blockers, matching people to problems, and staying hands-on technically so you understand the work.
