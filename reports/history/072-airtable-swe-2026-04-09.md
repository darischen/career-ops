## Interview Prep: STAR+R Stories

### Story 1: Full-Stack Ownership Under Pressure
**Situation:** PatentIQ needed to ship semantic search MVP in 4 months for a accelerator demo.
**Task:** I was the primary architect and engineer responsible for the entire stack.
**Action:**
- Designed API (FastAPI) for patent search and embedding retrieval
- Built frontend (Next.js) with real-time search and result ranking
- Chose PostgreSQL + pgvector for semantic search (no external vector DB)
- Implemented Auth0 for security, deployed to Vercel/EC2/Supabase
- Coordinated with 14-person team (data scientist, backend engineer, PMs)
**Result:** Shipped working product that early-stage founders use for professional patent research. ~40% improvement in discovery vs. full-text search alone.
**Reflection:** I learned that real constraints (budget, time, team size) matter more than perfect architecture. We shipped PostgreSQL + pgvector instead of waiting for a "better" vector DB. That pragmatism is what matters in production.

### Story 2: API Design for Scalability
**Situation:** Groundwork Books needed real-time inventory sync across 4,000+ products via Square API.
**Task:** Design an API layer that wouldn't hit Square's rate limits while keeping data fresh.
**Action:**
- Built an LRU cache in Redis that reduced API calls by 90%
- Implemented client-side request coalescing: batched 50+ concurrent lookups into one API call
- Added smart TTL (2-minute cache for inventory, longer for static data)
- Monitored API latency (target: sub-200ms p99)
**Result:** Achieved sub-200ms p99 latency, zero out-of-stock errors due to stale data
**Reflection:** Scalability isn't about the language or framework—it's about understanding your constraints (API limits, latency budgets) and designing within them.
