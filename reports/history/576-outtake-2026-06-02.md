**Reflection:** "Speed comes from pragmatism. I could've spent 6 hours debugging SAM3D on ROCm, or I could move to CUDA and ship. I chose to ship. That's the mentality I bring: solve the problem in front of you with what you have, not the perfect problem with hypothetical tools."

---

### Story 2: Owned Architectural Decisions (PatentIQ)

**Situation:** PatentIQ MVP. Design the semantic search backend for 8M patent dataset. PMs proposed MongoDB + Pinecone (expensive, vendor lock-in) or other options.

**Task:** Recommend and build the database + embedding layer.

**Action:**
- Analyzed trade-offs: Postgres + pgvector costs $0, Pinecone costs $400/mo. MVP doesn't need Pinecone yet.
- Built hybrid search: keyword search (Postgres BM25) + semantic re-rank (pgvector cosine similarity)
- Implemented recursive CTEs for patent family queries, optimized chunk size for embedding quality
- Documented decision in RFC for team review

**Result:** Shipped semantic search in 4 weeks instead of 8. Lower cost ($0 vs $400/mo). Easier to iterate on schema and embedding strategy.

**Reflection:** "Good architecture isn't about the fanciest tools. It's about understanding your constraints (cost, velocity, data scale) and making the decision that buys you the most options. Postgres + pgvector let us iterate faster, and if we needed Pinecone later, we could migrate. You ship with what lets you learn, not with what the benchmarks say is best."

---

### Story 3: Systems Thinking (Groundwork Books)

**Situation:** Groundwork's e-commerce platform getting slower as inventory data grew. Square API was the bottleneck.

**Task:** Design caching strategy to handle 4,000+ SKUs with real-time inventory.

**Action:**
- Analyzed bottleneck: Square API calls for inventory lookup on every product page load. 50+ concurrent requests on busy days.
- Implemented two-tier caching:
  - Backend: Redis cache (TTL 2 min) reduces Square API calls by 90%
  - Frontend: Request coalescing (batch 50+ concurrent lookups into 1 API call)
- Result: p99 latency dropped from 2s to sub-200ms

**Result:** Inventory updates in real-time (2-minute eventual consistency), zero user-facing slowdowns.

**Reflection:** "Systems thinking is understanding the whole flow: user request → API call → cache → database. The bottleneck wasn't the database, it was the API. The fix wasn't more servers, it was a smarter request strategy. That's how you build products that scale without chaos."

---

### Story 4: Team Leadership & Delegation (PatentIQ)

**Situation:** Lead architect for PatentIQ MVP. 14-person team: 2 PMs, 2 designers, 5 engineers, 5 data + ops.

**Task:** Deliver working product in 4 months. Clear ownership + minimal rework.

**Action:**
- Defined system architecture upfront (FastAPI + Auth0 + Supabase + Next.js), documented decisions
- Matched people to problems: data scientist owns ML pipeline end-to-end, backend engineer owns API, frontend engineer owns UI
- Held weekly syncs, resolved blockers (Auth0 SSO setup, Claude API rate limiting, pgvector query optimization)
- Reviewed code from each domain, advocated for small decisions (e.g., "let's chunk patents by section, not by word count")

**Result:** Shipped MVP with clear ownership. Each engineer knew their domain. Minimal rework (2 refactors total, both planned).

**Reflection:** "Leadership isn't about having all the answers. It's about creating clarity (who owns what, what's the goal, what's the trade-off) and trusting people to solve their domain. I didn't write every line of code, but I made sure the architecture let people move fast and didn't create blockers."

---

### Story 5: Pragmatism Over Perfection (FlipperZillow SAM3D decision)

**Situation:** GPU pipeline bottleneck. SAM3D (3D object extraction) incompatible with ROCm. Only one GPU instance at a time (expensive).

**Task:** Unblock the pipeline without blowing the hackathon budget.

**Action:**
- Diagnosed: SAM3D breaking dependency. Options: (1) debug 6+ hours, (2) move to CUDA, (3) cut SAM3D, (4) expensive parallel GPU instances
- Made decision: Cut SAM3D, keep 3D reconstruction via depth estimation + WebSpatial. Simpler, faster.
- Shipped product without 3D object extraction layer

**Result:** Working product shipped on time with all core features. User experience was 95% of "perfect" version.

**Reflection:** "Perfectionism kills shipping. In a 24-hour hackathon, every hour spent debugging is an hour not spent building. I ask: 'What's the minimum I need to ship something real?' SAM3D was nice-to-have, not core. We shipped without it, and the product still worked. That's the mentality Outtake values: ship real, iterate fast."
