## F) Interview Prep

### Story 1: Critical Path Optimization (CLA Adder)

**Situation:** Digital design capstone. Build a 16-bit adder. Baseline: ripple carry (slow). Goal: minimize latency.

**Task:** Reduce critical path from O(N) to O(log N) via carry look-ahead.

**Action:**
- Analyzed ripple carry: each bit waits for carry from previous bit. Critical path is O(16).
- Designed carry look-ahead: parallelize carry computation. Generate carry signals independently, combine them.
- Used hierarchical design: 4-bit CLA blocks (2 levels), trade-off between latency and complexity.
- Validated corner cases: worst-case inputs (e.g., 0x7FFF + 0x0001) to ensure correctness.

**Result:** O(log N) critical path. Latency reduced to 4-5 gate delays instead of 16.

**Reflection:** Physical design is similar optimization. You have millions of gates. You minimize critical path while managing power and area. My CLA adder shows I understand the core trade-off.

---

### Story 2: RTL Timing Analysis (TriCore9)

**Situation:** TriCore9 pipeline design. Need to meet timing at a certain clock frequency (e.g., 100 MHz).

**Task:** Design 4-stage pipeline. Ensure no stage exceeds timing budget.

**Action:**
- Calculated timing budget per stage: if clock period is 10ns, each stage gets ~2.5ns (includes pipeline registers).
- Identified critical path in each stage: ALU (longest path), register fetch (shorter).
- Optimized ALU via logic minimization (fewer gate levels) and pipelining (split ALU into two stages if needed).
- Used Quartus to measure actual path delays. Adjusted design to meet timing.

**Result:** TriCore9 synthesis achieved target frequency with timing slack.

**Reflection:** This is physical design work. You understand timing budgets, critical paths, pipelining to meet timing. In IC design, you do the same thing at larger scale (millions of gates instead of hundreds).

---

### Story 3: Trade-off Analysis (Area vs. Latency)

**Situation:** CLA Adder design. Trade-off: more logic gates → faster (lower latency) but larger area.

**Task:** Decide between 1-level CLA (fastest, largest), 2-level CLA (moderate), or ripple carry (slowest, smallest).

**Action:**
- For a 16-bit adder:
  - Ripple carry: minimal gates, O(16) latency.
  - 1-level CLA: more gates (4×), O(2) latency (4 group generates, then one carry merge).
  - 2-level CLA: compromise, O(log 4) ≈ O(3) latency, reasonable gate count.
- Chose 2-level CLA: met performance requirements without excessive area.

**Result:** Balanced design. Faster than ripple carry, smaller than full CLA.

**Reflection:** Physical design is full of similar trade-offs. You minimize power for given frequency. You minimize area for given frequency. You minimize frequency for given power. Every choice has consequences.

### Story 4: Design Validation (TriCore9 Simulation)

**Situation:** TriCore9 synthesis. Need to validate that RTL design works correctly on FPGA (behavioral simulation isn't enough).

**Task:** Synthesize, simulate on ModelSim, validate test programs on FPGA.

**Action:**

- Validated that FPGA results matched simulation (exact match on test outputs).
- Tested corner cases: arithmetic overflow, branch prediction failures, cache misses.

**Result:** Confidence that RTL design worked correctly end-to-end.

**Reflection:** Physical design has similar validation. You simulate with post-synthesis netlist, measure actual delays, validate timing closure. I understand this flow.
