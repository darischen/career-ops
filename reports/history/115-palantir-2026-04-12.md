### STAR Stories for Palantir Infrastructure Interviews

#### Story 1: Database Design Under Constraints
**JD Requirement:** Backend infrastructure design, database systems

**STAR Structure:**
- **Situation:** PatentIQ needed fast semantic search over patent database for early-stage founders
- **Task:** Choose persistence layer (PostgreSQL vs. MongoDB vs. hosted vector DB like Pinecone/Weaviate)
- **Action:**
  - Analyzed requirements: need pgvector for semantic similarity, fast iterations on schema, MVP speed
  - Proposed PostgreSQL + pgvector (not Pinecone/Weaviate) because: already running, open-source, no vendor lock-in, cheaper iteration
  - Argued with PMs and data scientist using benchmarks (latency, cost, flexibility)
  - Implemented hybrid ranking: 70% semantic (pgvector) + 30% keyword matching
  - Tuned indices and query plans to achieve sub-2s p99 on 5M-record dataset
- **Result:** 
  - Ship faster than managed vector DB would allow
  - Lower operational cost (no Pinecone license)
  - Better product UX (hybrid ranking > pure semantic)

**Why this matters for Palantir:** Shows database architecture thinking, cost consciousness, ability to argue for technical approach, shipped real system under real constraints

---

#### Story 2: Production Performance Optimization
**JD Requirement:** Scalability, performance optimization, understanding of systems tradeoffs

**STAR Structure:**
- **Situation:** Groundwork Books e-commerce site hitting bottlenecks during peak traffic. Square API latency was 500ms+ per inventory lookup; 4,000+ SKUs in catalog
- **Task:** Make real-time inventory sync work at scale without overpaying for API calls
- **Action:**
  - Diagnosed bottleneck: every product view = 1 Square API call
  - Designed two-tier caching:
    - Tier 1: Look-aside Redis cache (cache hot items, TTL 5min)
    - Tier 2: Client-side request coalescing (batch 50 concurrent requests into 1 API call, 2min TTL)
  - Implemented cache invalidation on inventory change (webhook from Square)
  - Monitored p50/p95/p99 latencies continuously
- **Result:**
  - p99 latency: 2.1s → 200ms (90% reduction)
  - Reduced Square API calls by 95%
  - Maintained real-time consistency (max staleness = 5 min)

**Why this matters for Palantir:** Shows systems thinking (caching strategy), production reliability, ability to think about tradeoffs, shipped under real constraints

#### Story 4: Systems Thinking Across Layers
**JD Requirement:** Understanding of data structures, storage systems, optimization

**STAR Structure:**
- **Situation:** Mini-Stockfish chess engine: NNUE evaluation was bottleneck (1.1ms per leaf node evaluation)
- **Task:** Reduce latency so engine can search deeper and play competitively in real-time
- **Action:**
  - Profiled code: 70% of time in Python neural network evaluation
  - Attempted optimization path 1: vectorized NumPy — helped but not enough
  - Attempted optimization path 2: Cython compilation — 3x speedup
  - Attempted optimization path 3: implemented O(1) neural network accumulator (instead of recomputing evaluation after each move, incrementally update)
  - Added bitboard representation for move generation in C++
  - Zobrist hashing for transposition table fast lookups
- **Result:**
  - 77.2% latency reduction: 1.1ms → 0.25ms per node
  - Enables real-time competitive play (search 20+ plies in reasonable time)
  - Competitive strength trained on 316M Stockfish positions

**Why this matters for Palantir:** Shows ability to optimize from high-level (Python) down to low-level (C++ + bitboards). Infrastructure engineers need this thinking.
