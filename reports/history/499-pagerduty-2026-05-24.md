## Block F — Interview Prep

### Story 1: Async Systems at Scale (PatentIQ)
- **Situation:** Patent search MVP needs to handle concurrent requests while maintaining semantic search latency <2s. Architecture: FastAPI backend + pgvector database + LRU cache.
- **Task:** Design async pipeline that doesn't bottleneck on database or embeddings.
- **Action:** Used FastAPI's async/await, async database drivers (asyncpg), and request batching. Implemented LRU cache to avoid re-computing embeddings. Tuned connection pool to handle 50+ concurrent requests.
- **Result:** MVP handles concurrent load; p95 latency <2s. Users report instant search.
- **Reflection:** Async systems are about queueing, batching, and graceful degradation. Notifications are the same pattern. If the notification queue gets backed up, users should get notification, even if it's slightly delayed.

### Story 2: Multi-Channel API Integration (Groundwork Books)
- **Situation:** Groundwork Books needs to sync inventory from Square POS in real-time. Also: show community events from Instagram + Google Sheets. Different APIs, different rate limits, different data formats.
- **Task:** Integrate 3 APIs without overwhelming the database or frontend.
- **Action:** Built async jobs to poll Square (every 5min), Instagram (hourly), Google Sheets (hourly). Cached results in Upstash Redis. Frontend queries cache first; cache miss triggers async update in background.
- **Result:** Real-time inventory, event updates without friction. API costs cut 50%.
- **Reflection:** PagerDuty has 100+ channels; same problem. You need to abstract the channel interface (email, SMS, Slack have different APIs, rate limits, failure modes). Build once, integrate many times.

### Story 3: Pragmatism Under Pressure (FlipperZillow)
- **Situation:** 24-hour hackathon. GPU pipeline failing on AMD ROCm. Debugging would take 3+ hours.
- **Task:** Ship working 3D house tours in 24 hours.
- **Action:** Recognized the cost of debugging. Pivoted to NVIDIA cloud instead of burning time on ROCm compatibility. Lost some efficiency, won 8 hours of dev time.
- **Result:** Shipped working product on time.
- **Reflection:** In on-call situations, pragmatism wins. If a notification channel goes down, you route around it, not debug it live at 3am. Then fix it during business hours.

### Story 4: Reliable Systems (PatentIQ)
- **Situation:** PatentIQ MVP went live. Founders using it for professional research. If search fails, their work stops.
- **Task:** Ensure high availability (no cascade failures).
- **Action:** Implemented fallback: if pgvector search fails, fall back to keyword-only search (slower, but works). Monitored error rates via Vercel/EC2 dashboards. On-call rotation (before formal job): I monitored it.
- **Result:** 99.5% uptime during accelerator. Zero unplanned downtime.
- **Reflection:** Notifications demand similar reliability. Users expect messages to arrive reliably, even under load. This means understanding failure modes and building fallbacks.

### Story 5: End-to-End Feature Ownership (Simplerfy)
- **Situation:** Simplerfy needs resume generation + LaTeX compilation + PDF delivery. Three different systems.
- **Task:** Ship end-to-end pipeline from job description → tailored resume → PDF.
- **Action:** Designed the interface: Claude API returns structured JSON. Backend compiles LaTeX. Frontend downloads PDF. Tested the full pipeline before shipping.
- **Result:** Users get PDF in <5s.
- **Reflection:** At PagerDuty, you'll own similar end-to-end features: user configures notification rule → rule triggers → message routed to channel → delivered. Each layer matters.
