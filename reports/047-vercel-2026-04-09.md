### Case Study / Demo Strategy

**Recommended project to present:** **Groundwork Books Caching Architecture**

**Why this project:**
1. **Scale:** Real users, real traffic, real constraints (Square API limitations)
2. **Reliability:** 90% latency improvement is a concrete business outcome
3. **Systems thinking:** Two-tier solution shows maturity
4. **Live product:** Can show actual site running fast
5. **Technical depth:** Can discuss Redis internals, request batching, cache invalidation trade-offs

**How to present (30-min walkthrough):**

1. **Problem statement (2 min):** "Square POS API is the bottleneck. 15+ concurrent inventory checks per user. No batch endpoint. p99 latency: 8s. Impact: users churn."

2. **Constraints (2 min):** "Can't change Square API. Can't add infrastructure cost. Can't lose consistency (must reflect real-time inventory)."

3. **Solution architecture (5 min):**
   - Diagram: Square API → Redis (look-aside) + PostgreSQL (source of truth) + Client-side coalescing
   - Walk through a user flow: user clicks product → multiple `<InventoryCheck>` components all query at once → request coalescing hook batches them → single API call to Square → result cached in Redis for 2 min → subsequent users hit cache

4. **Implementation (5 min):**
   - Show code: React hook `useInventoryCache()` with debounce + cache
   - Show Redis configuration: TTL strategy, eviction policy
   - Show invalidation logic: when cart is updated, purge stale cache keys

5. **Results (2 min):**
   - Metrics: p99 latency (8s → 200ms), API call volume (90% reduction), user retention (correlated improvement)
   - Live demo: Load test with 100 concurrent users, show sub-300ms response time

6. **What you'd do differently at Vercel (3 min):**
   - "At Vercel scale, I'd think about: (1) Cache coherency across regions (CloudFlare vs. CDN?), (2) Observability (metrics on cache hit rate, eviction), (3) Cost optimization (does this caching strategy scale to 10,000+ SKUs?)"
